# Internationalisation (i18n)

Read this when adding or changing any user-facing copy, or working with locale,
dictionaries, or `Localized` data.

The site is bilingual **Danish (default) / English**, switched by the header
toggle. There is **no locale routing**. The choice is stored in a cookie
(`aff_locale`) and read server-side, so the URL never changes.

- **UI-chrome strings** (buttons, headings, aria-labels, alt text) live in
  `src/i18n/dictionaries.ts` as `{ da, en }`. The Danish object defines the
  `Dictionary` shape; English must match it (missing keys = type error).
  Interpolated values (titles, dates) are composed at the call site. Shared
  nav/chrome labels belong here even when a page renders them — e.g. the
  back-to-home button reads `t.common.backToHome`, not a per-page `backLabel` in
  `src/data/`. Keep such repeated chrome out of the content data.
- **Content data** in `src/data/` stores each translatable field as
  `Localized<string>` (`= { da, en }`); structural fields (`id`, `slug`, `href`,
  `time`, `dayId`, `tone`, `icon`) stay plain. Proper nouns that don't translate
  stay plain too: an artist `name` is a `string`, while a descriptive workshop
  `name` is `Localized`.
- **Server components** read the locale with `await getLocale()` from
  `@/i18n/server` and `getDictionary(locale)`, then index `field[locale]`. Cards
  receive `locale` as a prop.
- **Client components** use `useLocale()` / `useTranslations()` from
  `@/i18n/LocaleProvider`. Setting a locale writes the cookie and calls
  `router.refresh()` so the server tree re-renders.
- Because the server reads `aff_locale`, public routes are currently dynamic and
  may miss static-page caching/bfcache optimisations. Keep this tradeoff unless
  the locale architecture changes (for example locale routes or a client-only
  locale shell).

When adding copy: add the key to **both** `da` and `en` (or a `Localized` field
to the data), never hardcode a user-facing string in a component.

> For the voice and wording rules that govern the copy itself, see
> [content-and-tone.md](content-and-tone.md).
